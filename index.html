How I Work

Problem framing I start from data and signal, not assumptions—gap analysis, cycle data, or direct stakeholder input before scoping any solution
Design philosophy Reinforce existing organizational language and frameworks where possible; reduce cognitive load, not add to it
Build quality Production-ready systems, not prototypes—built to be maintained by non-technical stakeholders
Measurement Every program has a diagnostic hook—pilot data, participation metrics, or behavioral observation frameworks
Initiative Several of these projects were self-initiated—I identified the problem, proposed the solution, and built it without a formal request

How I Work

Problem framing I start from data and signal, not assumptions—gap analysis, cycle data, or direct stakeholder input before scoping any solution
Design philosophy Reinforce existing language and frameworks where possible; reduce cognitive load, not add to it
Build quality Production-ready systems, not prototypes—built to run without me maintaining them
Measurement Every program has a diagnostic hook—pilot data, participation metrics, or behavioral observation frameworks
Systems thinking I build the infrastructure first—templates, automation, architecture—then leverage it. The same structural patterns show up in my curriculum, my code, and my creative work
Same skill, different medium Prompt engineering, learning design, music production, memory architecture—different outputs, identical process: define a base, apply contextual modifiers, iterate within constraints
